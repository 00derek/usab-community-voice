# Contributing to USAB Community Voice

Thank you for helping document and improve issues that matter to the badminton community. This repo is a shared space — contributions from players, parents, coaches, club leaders, officials, and USAB members are all welcome.

## How to Contribute

### Submitting feedback

The easiest way to share your thoughts is through the **[community feedback form](https://docs.google.com/forms/d/e/1FAIpQLScrp_rzg7bhIJdpnF74Te7dnxWacTYk8plZI045A96DvILNgQ/viewform)**. No GitHub account required. You can select which issue you're commenting on, describe your experience or concern, and optionally leave your name and email for follow-up.

If you prefer to engage directly on GitHub, you can also:
- Open a **[GitHub Issue](https://github.com/00derek/usab-community-voice/issues/new)** to raise a specific concern or question
- Start a **[GitHub Discussion](https://github.com/00derek/usab-community-voice/discussions)** to explore something more openly with others

Maintainers review all submissions and can help shape recurring themes into proper issue write-ups.

### Improving an existing issue

- **Adding context**: If you have first-hand experience, data, or background that enriches the picture, add it to the Context section.
- **Proposing options**: Add new options to the Potential Options section. Remember — these are illustrative, not prescriptive.
- **Answering open questions**: If you know the answer to one of the Open Questions, update the issue or note it in your PR description so the question can be refined or removed.
- **Correcting inaccuracies**: If something is factually wrong, please correct it and note the change in your PR.

## Writing Guidelines

- **Be specific**: Vague concerns are hard to act on. Name the event, the policy, the year, the number — concrete details make issues stronger.
- **Be constructive**: The goal is to surface problems clearly enough that solutions become possible. Avoid personal criticism of individuals.
- **Be honest about uncertainty**: If you're not sure of a fact, say so. "Reportedly" or "approximately" is better than presenting uncertain information as confirmed.
- **Separate problems from solutions**: The Issue and Context sections describe what is wrong. The Potential Options section explores what could be done. Keeping these distinct makes issues easier to evaluate.

## Questions

Open a GitHub Issue if you're unsure whether something belongs here or how to frame it. We'll figure it out together.
